# 智能库存分配系统 V35.8

基于 Streamlit 构建的跨境电商智能库存调拨系统。针对多仓、多标（FNSKU）、多渠道的复杂库存场景，实现从数据清洗到智能分配的全链路自动化。

## 技术栈

- **前端**：Streamlit
- **数据处理**：Pandas
- **导出**：xlsxwriter

## 快速启动

```bash
pip install streamlit pandas xlsxwriter
streamlit run app.py
```

## 系统输入

| 文件 | 说明 | 必填 |
|------|------|------|
| 需求表 | 在页面表格中直接填写或粘贴，包含标签、国家、SKU、FNSKU、数量等列 | 是 |
| A. 库存表 | 在库现货数据（支持 xlsx / csv） | 是 |
| B. 采购追踪表 | 采购订单 / 在途 PO 数据 | 是 |
| C. 提货计划表 | 提货计划数据 | 否 |

系统内置智能表头识别（`load_and_find_header`），支持非标准格式的 Excel 和 CSV（含 GBK 编码自动回退），无需手工调整源文件格式。

## 模块一：底层数据净化与建池（InventoryManager 类）

这是系统能够做到"账实相符"的基石。在进行任何分配之前，代码会先对上传的三个文件进行**脱水处理**。

### 1.1 仓库黑名单隔离

在载入库存表时（`_init_inventory`），系统会硬性过滤掉包含 `沃尔玛`、`WALMART`、`TEMU` 等字眼的仓库记录，将其排除在常规分配池之外。每条被过滤的记录都会写入清洗诊断日志，便于事后审计。

```python
# app.py:134
if any(k in w_name_norm for k in ["沃尔玛", "WALMART", "TEMU"]):
    self.cleaning_logs.append({"类型": "库存过滤", "SKU": sku, "原因": f"剔除黑名单仓库 ({w_name_raw})"})
    continue
```

### 1.2 橡皮擦机制（提货计划 扣减 PO）

**业务痛点**：提货计划在现实中是已经发货在途的采购订单（PO）。如果不处理，会导致这批货被**计算两次**——一次出现在提货计划表，一次出现在采购追踪表的"未入库"列中。

**实现逻辑**（`_deduct_plan_from_po`）：

```
提货计划 ──▶ 橡皮擦 ──▶ 扣减 PO 虚假未入库量
```

1. **精准扣减**：优先匹配同 SKU + 同 FNSKU 的 PO 记录，逐笔抵消
2. **兜底扣减**：若精准扣减后仍有剩余，则跨越 FNSKU 限制，对同 SKU 下其他条码的通货 PO 进行强行扣减

```python
# app.py:195-213
# 第一轮：精准扣减（同SKU + 同FNSKU）
if plan_fnsku in self.po[sku]:
    for po_item in self.po[sku][plan_fnsku]:
        take = min(po_item['qty'], qty_to_deduct)
        po_item['qty'] -= take
        qty_to_deduct -= take

# 第二轮：兜底扣减（跨FNSKU）
if qty_to_deduct > 0:
    for other_fnsku, po_items in self.po[sku].items():
        # ...强行扣减
```

### 1.3 重组资源池（`_merge_inbound_for_allocation`）

清洗完成后，系统将提货计划和挤干水分的净 PO 统一装入 `inbound`（在途供应池），供后续分配引擎调用。其中 PO 仅保留 `qty > 0` 的有效记录（已被提货计划擦干净的部分不再参与分配）。

### 1.4 数据结构设计

系统采用 **三层嵌套字典** 作为核心数据结构，实现 SKU 级别的精细化库存管理：

```
stock[SKU][FNSKU][仓库类型] = [{'qty': 数量, 'raw_name': 原始仓库名, 'zone': 库位}]

仓库类型枚举：深仓 | 外协 | 云仓 | 其他
```

仓库归一化规则（`normalize_wh_name`）：

| 关键字 | 归类 |
|--------|------|
| 包含"深" | 深仓 |
| 包含"外协" | 外协 |
| 包含"云"或"天源" | 云仓 |
| 其他 | 不参与常规分配 |

## 模块二：上帝视角与任务统筹（run_allocation 前半部分）

在正式开始扣库存之前，系统会做两项全局性统筹工作。

### 2.1 SKU 级缺口预判（生成待下单清单）

系统将所有需求按 SKU 汇总，对比全局总供应量（国内现货 + 提货计划 + 净PO）：

```
总需求 vs 总有效供应 = Gap
Gap > 0 → 生成《待下单清单》
```

输出的待下单清单包含以下字段，指导采购部进行真实补单操作：

| 字段 | 说明 |
|------|------|
| SKU | 产品编码 |
| 总需求 | 该 SKU 所有需求行汇总 |
| 国内库存(深+外+云) | 现货总量 |
| 提货计划总量 | 在途提货计划 |
| 净PO未入库(已清洗) | 经过橡皮擦去重后的真实 PO 量 |
| 总有效供应 | 以上三项合计 |
| 建议补单缺口 | 总需求 - 总有效供应 |

### 2.2 SJF 短作业优先排序（防饿死机制）

将所有需求行放入 `tasks` 列表，并严格按需求数量（qty）**从小到大**升序排序：

```python
# app.py:399
tasks.sort(key=lambda x: x['qty'])
```

**意义**：在库存有限的情况下，优先满足数量小的订单。这样可以用最少的库存解决最多的订单行，极大提高整体的**订单满足率**，避免大单独占库存导致众多小单全部缺货的"饿死"问题。

## 模块三：核心分配引擎（4 大贯通阶段）

这是整套代码最精密的部分。所有订单必须**跑完当前阶段**，才能进入下一阶段。同一阶段内，所有订单共享当时的库存快照，确保分配结果的一致性。

### 分配模式说明

引擎通过 `mode` 参数控制匹配严格程度：

| 模式 | 说明 | 适用阶段 |
|------|------|----------|
| `strict_only` | 仅匹配 FNSKU 完全一致的货 | R0、R1 |
| `process_only` | 跨越 FNSKU，匹配异标货（撕标换标） | R2、R3 |

---

### 阶段 0：US 独享智能防爆仓 / 单仓整发

**适用对象**：仅限美国站（US）订单。

**核心目标**：节省库管跨区调拨和打包的人工成本，优先尝试"一口吃饱"的单节点整发。

**执行流程**：

```
                     ┌─────────────────────────┐
                     │   US 订单进入阶段 0       │
                     └────────────┬────────────┘
                                  │
                    ┌─────────────▼──────────────┐
                    │ 依次探查4个现货节点          │
                    │ 外协→云仓→提货计划→深仓     │
                    └─────────────┬──────────────┘
                          ┌───────┴───────┐
                          │               │
                    任一节点 ≥ 需求    4个节点都不够
                          │               │
                    单节点整发       ┌─────▼──────────┐
                    （结束分配）     │ 净PO 能一口吃满足？│
                                    └─────┬──────────┘
                                      ┌───┴───┐
                                     是       否
                                      │       │
                            ┌─────────▼─┐   跳过，交给后续阶段
                            │ 防爆仓拦截 │
                            └─────┬─────┘
                           ┌──────┴──────┐
                     差距 ≤ 200      差距 > 200
                           │              │
                    清空现货最大节点    PO 直接整发
                    剩余交给后续阶段   （结束分配）
```

**防爆仓拦截详解**：

当前 4 个现货节点都不够单节点整发，但净 PO 够一口吃饱时，系统会回头找出现货最多的那个节点（例如深仓有 400 个，需求是 500 个）：

- **差距 ≤ 200**：为了消耗国内现货、防止滞销爆仓，系统**拒绝**净 PO 整发，强行清空该现货节点（400 个），剩余 100 个缺口交给后续阶段拼凑
- **差距 > 200**：拼凑成本太高，放过现货，让净 PO 直接整发

```python
# app.py:440
if max_qty > 0 and (t['qty'] - max_qty) <= 200:
    # 清空现货最大节点，防爆仓
    r, u, p, l, eu = inv_mgr.execute_deduction(t['sku'], t['fnsku'], max_qty, [max_node], 'strict_only')
```

### 阶段 1：全通道精准贯通刮肉（绝对不撕标）

**适用对象**：全体未满订单（US + 非US）。

**逻辑**：在资源池中寻找 FNSKU 完美匹配的货，按固定链路依次刮取，一刮到底。

**刮肉链路**：

```
US 订单：  外协 → 云仓 → 提货计划 → 深仓 → 净PO
           （优先消耗外面的货，保全深仓）

非US 订单：深仓 → 外协 → 云仓 → 提货计划 → 净PO
           （深仓直发成本最低，优先使用）
```

```python
# app.py:451-452
strat = [('stock', '外协'), ('stock', '云仓'), ('inbound', '提货计划'), ('stock', '深仓'), ('inbound', '采购订单')] if t['is_us'] else \
        [('stock', '深仓'), ('stock', '外协'), ('stock', '云仓'), ('inbound', '提货计划'), ('inbound', '采购订单')]
```

---

### 阶段 2：非 US 独享异标加工（撕标换标）

**适用对象**：仅限非美国站订单（美国站换标成本极高，禁止参与）。

**逻辑**：如果阶段 1 没吃饱，允许跨越 FNSKU 限制，去同 SKU 下其他 FNSKU 的货来顶替。系统记录为"加工"操作，输出加工库区、库位、原始 FNSKU 等信息供仓库执行撕标贴标。

**刮肉链路**（均为异标匹配）：

```
深仓(异标) → 外协(异标) → 云仓(异标) → 提货计划(异标)
```

```python
# app.py:461
strat = [('stock', '深仓'), ('stock', '外协'), ('stock', '云仓'), ('inbound', '提货计划')]
r, u, p, l, eu = inv_mgr.execute_deduction(t['sku'], t['fnsku'], rem, strat, 'process_only')
```

---

### 阶段 3：全局净 PO 兜底盲配

**适用对象**：全体仍有缺口的订单。

**逻辑**：最后的底牌。在净 PO 中寻找跨 FNSKU 的记录（未指定条码或标注"通用/新品"等），强行盲配扣减完成补足。

```python
# app.py:469-471
strat = [('inbound', '采购订单')]
r, u, p, l, eu = inv_mgr.execute_deduction(t['sku'], t['fnsku'], rem, strat, 'process_only')
```

## 模块四：底层字典聚合与前端输出（数据拼装）

经过多轮"榨干式"分配后，系统通过底层聚合算法将零碎的分配动作转化为指导实际业务的报表。

### 4.1 多轮次深度聚合（update_task）

这是数据拼装的核心函数。无论一个订单跨越了多少个仓库、经历了多少个分配轮次，`usage` 和 `entity_usage` 字典都通过 `.get(k, 0) + v` 进行**不断累加，绝不覆盖**：

```python
# app.py:406-407
for k, v in usage.items():
    t['usage'][k] = t['usage'].get(k, 0) + v
```

最终在前台完美拼接出类似 `外协仓库存500+深仓库存200` 的全景库存状态。

### 4.2 发货主体溯源（entity_usage）

精确提取每一笔扣减操作背后的**原生仓库名称**（如"深圳市德拉姆供应链有限公司"），输出至专列，为财务核算不同主体的发货成本提供依据。

### 4.3 精准调拨指令

计算非 US 订单中占用的外协和云仓数量，输出在 `需调回深仓数量(外协/云仓)` 专列，指导实际物流调拨：

```python
# app.py:511
waixie_transfer_qty = to_int(t['usage'].get('外协', 0) + t['usage'].get('云仓', 0)) if not t['is_us'] else 0
```

### 4.4 应急备选参考（防万一透视眼）

系统会后台计算该 SKU **其他异标现货**的总量（排除当前 FNSKU），输出在 `同SKU其他现货参考` 列。当业务员面临缺货急需发货时，可据此作为线下特批强行撕标发货的数据支撑。

```python
# app.py:258-263  get_other_fnsku_stock
for f in self.stock[sku]:
    if f != current_fnsku:
        total += sum(i['qty'] for w in self.stock[sku][f] for i in self.stock[sku][f][w])
```

### 4.5 输出报表一览

| Sheet | 内容 |
|-------|------|
| 分配结果 | 每行需求的完整分配方案（发货主体、库存状态、最终数量、调拨提示等） |
| 待下单清单(已去重) | Gap > 0 的 SKU 汇总，指导采购补单 |
| 运算日志 | 每行需求的逐阶段执行过程记录 |
| 清洗去重日志 | 库存过滤、PO 扣减等底层清洗动作记录 |

## 列映射机制

系统通过 `_match_col` 实现模糊列名匹配，支持不同格式的源文件：

| 逻辑字段 | 可识别的列名关键字 |
|----------|-------------------|
| SKU | `SKU`、`编码`、`代码`、`型号` |
| FNSKU | `FNSKU`、`条码`、`标签`、`贴标要求` |
| 仓库 | `仓库` |
| 库位 | `库位`、`库区`、`ZONE` |
| 数量 | `可用`、`数量`、`库存`、`未入库`、`未交`、`在途` |

页面上的列映射配置下拉框可手动覆盖自动识别结果。
